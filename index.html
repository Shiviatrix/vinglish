The Englist Specification

Comprehensive Language Guide and Compiler Internals

This document serves as the exhaustive, unabridged specification of the Englist programming language, its standard library, and the internal architecture of its Rust-based compiler. It is designed to guide developers from the surface-level syntax down to the bare metal representation in the C-codegen backend.

1. Motivation & History

Systems programming languages typically fall into two camps: low-level, highly symbolic languages (C, C++) that offer ultimate control but sacrifice human readability, or modern, safe languages (Rust) that introduce immense cognitive overhead and complex syntax. Englist bridges this divide.

The core philosophy of Englist is: Code should read like English prose, but the compiler should rigorously understand and optimize the developer's intent. By parsing naturalistic keywords (is below, repeat while, be) and feeding them into an aggressively optimizing, mathematically pure Static Single Assignment (SSA) pipeline, Englist offers the readability of pseudocode with the zero-cost abstractions of C.

2. Lexical Structure

Englist programs are composed of a stream of ASCII tokens. The lexer ignores whitespace (spaces, tabs, newlines) outside of string literals.

Keywords

The following keywords are reserved and carry semantic weight:

public, private, package, module, use, fn, function, returns, begin, end,
if, then, else, repeat, while, let, be, is, below, above, borrow, mutable,
struct, type, address

Identifiers & Comments

Identifiers must begin with an alphabetic character or underscore, followed by alphanumeric characters or underscores. Line comments begin with // and block comments are currently unsupported to encourage concise, single-line annotations.

3. Types & Semantics

Englist employs a static, strongly-typed system. All variables, function parameters, and return types must be resolvable at compile-time during the High-Level Intermediate Representation (HIR) phase.

Primitive Types

  • number: Represents the default architecture-dependent integer (equivalent to int64_t on 64-bit platforms).
  • address<T>: A raw, unmanaged memory pointer pointing to a value of type T. Equivalent to T* in C.
  • boolean: Logical true or false. (Implemented natively as a 1-byte integer).

Type Definitions

Types can be aliased using the type keyword, heavily used when defining generic structural boundaries.

4. Control Flow

Control flow in Englist is designed to read rhythmically, avoiding the "bracket fatigue" common in C-family languages. The begin and end keywords delineate scope.

Conditional Logic

The if / then / else construct supports relational operators modeled on English:

  • is (equality: ==)
  • is below (less than: <)
  • is above (greater than: >)
if x is below 10 then
    // Executed if x < 10
    x be x + 1
else
    // Executed otherwise
    x be 0
end

Iteration

Looping is handled via the repeat while construct. Englist does not currently feature a traditional for loop, preferring explicit index management or iterator patterns defined in the standard library.

let i be 0
repeat while i is below 100
    // Loop body
    i be i + 1
end

5. Memory & Ownership

Englist does not include a Garbage Collector (GC). The language requires the developer to manage memory allocations manually, but introduces syntactical safety barriers via the borrow system.

Manual Allocation Risk

Failure to invoke eng_free on an address initialized via eng_alloc will result in a memory leak. The standard library encapsulates these raw calls into safe data structures like Vector<T>.

Borrow Semantics

When passing structures across function boundaries, Englist enforces ownership clarity:

  • borrow: Passes an immutable reference to the function. The function can read, but the HIR compiler will reject any assignment to the borrowed data.
  • borrow mutable: Passes a mutable reference. The function may alter the underlying data.
public function transform(borrow mutable Vector<number> v)
begin
    // 'v' can be modified here.
end

6. Generics & Polymorphism

Englist achieves polymorphism through compile-time monomorphization of generics. A generic type parameter <T> acts as a placeholder during the parsing and initial HIR phases.

When a generic function or struct is instantiated (e.g., Vector<number>), the compiler generates a specialized, concrete version of that entity. This completely eliminates dynamic dispatch overhead at runtime.

struct Pair<T>
    first: T
    second: T
end

function make_pair<T>(a: T, b: T) returns Pair<T>
begin
    let p be Pair<T> { first: a, second: b }
    return p
end

7. Error Handling

Englist avoids exceptions entirely, utilizing explicit Result<T, E> types combined with the postfix ? operator for safe error propagation. Errors must be handled or explicitly propagated, ensuring resilient code.

public function read_config() returns Result<text, text>
begin
    let file be open("config.txt")?
    let content be read_to_string(borrow file)?
    return Ok(content)
end

8. String Formatting

Dynamic string interpolation is handled natively at compile time via the fmt! macro. It parses the format string to evaluate expressions and automatically coerces primitive types into a safe textual representation.

public function greet(name: text, age: number)
begin
    let msg be fmt!("Hello, my name is {name} and I am {age} years old.")
    print(msg)
end

9. Structs & Layout

Structs in Englist map 1:1 with C structures. They guarantee memory layout order, making them ideal for systems programming, foreign function interfacing (FFI), and network packet serialization.

Struct instantiation utilizes brace syntax. Every field must be initialized.

8. Library Overview

The Englist standard library (std) is a collection of modules written entirely in Englist. Because the language supports raw memory operations and seamless C interoperability, almost no internal C code is required. The library proves the language's capability to bootstrap itself.

9. Memory Allocation

The std.runtime module exposes intrinsics mapped directly to OS-level memory allocators (via the rt.c runtime wrapper).

  • eng_alloc(bytes: number) -> address<T>: Allocates uninitialized heap memory.
  • eng_free(ptr: address<T>): Frees the previously allocated memory block.
  • eng_read(ptr: address<T>, offset: number) -> T: Reads a value at a specific index offset.
  • eng_write(ptr: address<T>, offset: number, val: T): Writes a value to a specific index.

10. Dynamic Vectors

The Vector<T> is the quintessential dynamically resizing contiguous array. Internally, it holds a raw address, a length, and a capacity.

Reallocation Strategy

When vec.len is vec.capacity, the push function calculates new_cap = capacity * 2, allocates a new block, copies existing elements via a repeat while loop, and frees the old block. This guarantees O(1) amortized insertion time.

use std.collections.vector

function main()
begin
    let my_vec be vector_new<number>()
    vector_push(my_vec, 100)
    vector_push(my_vec, 200)
    
    // Safe deallocation
    vector_free(my_vec)
end

11. Hash Maps (Planned)

The upcoming HashMap<K, V> utilizes an open-addressing, linear-probing architecture to maximize CPU cache locality. The hashing algorithm employs a fast, non-cryptographic hash (such as FNV-1a or WyHash), resolving collisions by scanning adjacent memory slots.

12. File I/O

The std.io and std.fs modules provide wrappers around POSIX file descriptors. Files are represented as structs containing the integer descriptor and buffer pointers, enabling buffered reads and writes natively in Englist.

13. The Pipeline

The Englist compiler, built in Rust, executes a strict, unidirectional pipeline. Each stage strictly enforces mathematical and logical correctness before handing off to the next layer.

Source Code (.eng) 
       ↓ (Lexer & Parser)
Abstract Syntax Tree (AST)
       ↓ (Semantic Analysis & Type Checking)
High-Level IR (HIR)
       ↓ (SSA Conversion & CFG Generation)
Mid-Level IR (MIR)
       ↓ (Optimization Passes: DCE, GVN, Constant Fold)
Optimized MIR
       ↓ (Codegen Backend)
C Source Code (.c)
       ↓ (GCC / Clang)
Executable Binary

14. Lexer & Parser

The Lexer converts raw text into a token stream. The Parser utilizes a recursive descent algorithm with Pratt parsing (Operator-Precedence parsing) for expressions. This accurately handles operator precedence, converting English keywords (like is below) into strict binary operations within the AST.

15. High-Level IR (HIR)

The AST is flattened into the HIR. This is where Symbol Resolution occurs. The compiler maps all identifiers to unique SymbolIds. Module imports (use std.math) are resolved into an Import Graph, allowing the compiler to selectively merge required definitions and detect namespace collisions.

Alias Analysis is also performed here. By tracking pointer assignments and references, the compiler validates that mutations do not violate borrowing rules.

16. Mid-Level IR & SSA

The HIR is translated into the Mid-Level IR (MIR). This is the crown jewel of the Englist compiler.

MIR utilizes Static Single Assignment (SSA) form. In SSA, every variable is assigned exactly once. When a variable is reassigned in the source (e.g., x = x + 1), the MIR generates a new version of that variable (e.g., x_2 = x_1 + 1).

Phi (Φ) Nodes

At control flow merge points (e.g., the end of an if/else block), the MIR inserts a Phi node. The Phi node selects the correct version of a variable based on which basic block executed previously (e.g., x_3 = Φ(x_1, x_2)). This pure mathematical mapping allows optimization passes to trace data-flow flawlessly.

17. Opt: CFG Simplification

The cfg_simplify.rs pass analyzes the Basic Blocks of the MIR. It performs:

  • Dead Block Elimination: Removes blocks that have no incoming edges.
  • Block Merging: If Block A unconditionally jumps to Block B, and Block B has no other incoming edges, the two blocks are merged into a single contiguous block, reducing branch overhead.

18. Opt: GVN & Constant Fold

Global Value Numbering (GVN): The compiler assigns a unique hash to every computed expression. If it encounters the same expression (e.g., a * b) later in the control flow, and neither a nor b have been modified (guaranteed by SSA), it replaces the second computation with a direct reference to the first result.

Constant Folding (constant_folding.rs): Traverses the SSA graph identifying expressions composed entirely of literal constants. It evaluates these at compile-time (e.g., folding 5 * 4 directly into 20) and propagates the result downstream.

19. C Backend Codegen

The highly optimized MIR is lowered into C code via lower.rs. Compiling to C rather than LLVM IR directly allows Englist to remain highly portable and easy to debug while still leveraging the world-class optimization of GCC or Clang.

Structure Topological Sorting

C requires that structures are defined before they are used. If Struct A contains a Struct B by value, Struct B must be declared first. The Englist codegen performs a topological sort on the dependency graph of all structs to emit them in the mathematically correct order, resolving cyclic pointer references automatically via forward declarations.

__auto_type Inference

To elegantly handle the translation of Englist generics to C, the codegen utilizes the GNU C extension __auto_type. This prevents the compiler from needing to generate deeply convoluted generic C macros, instead relying on the C compiler to infer the type of the evaluated expression.

// Generated C code for generic assignment
__auto_type var_3 = eng_std_vector_push(var_1, var_2);

20. Englist CLI

The eng-cli binary provides the primary interface to the compiler toolkit. Currently, it supports the compile command, which accepts a root .eng file, resolves its dependency graph, invokes the internal Rust pipeline, links rt.c, and spawns the host C compiler.

21. Language Server (LSP) Support

Because the compiler retains an intact HIR that understands alias analysis and full symbol mapping, Englist is perfectly positioned for a Language Server Protocol implementation.

  • Hover: The SSA graph directly maps identifiers to their inferred types.
  • Diagnostics: Borrow-checking errors and semantic validation are emitted as JSON LSP diagnostics.
  • Code Actions: The optimizer's knowledge of Dead Code and Redundant computations can be exposed as automated quick-fixes.

22. Future Roadmap

The roadmap for Englist prioritizes ecosystem expansion over language mutations:

  1. The Package Manager (eng): A native dependency resolver allowing commands like eng new, eng build, eng add std.net.
  2. Standard Library Maturation: Implementing cryptographic hashing, highly concurrent threading models, and a non-blocking TCP server entirely natively.
  3. LLVM Backend: While the C backend is excellent for bootstrapping, a native LLVM IR backend is planned for even tighter optimization control and faster compilation speeds (bypassing the C compiler intermediary).

23. Benchmarks

The Englist compiler includes an automated benchmarking suite to continually monitor the performance of the generated code against other mainstream languages like Rust, C++, and Go.

You can execute the benchmarking suite using the built-in CLI command:

eng benchmark benchmarks --runs 5

The performance metrics are tracked daily through our continuous integration pipelines, ensuring the C and LLVM codegen backends maintain zero-cost abstraction guarantees.
