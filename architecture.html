01 / PIPELINE

One explicit path.

.ving source
→ lexer + parser → AST
→ semantic analysis → HIR
→ MIR lowering → SSA MIR
→ optimization → MIR C backend
→ C + runtime → native binary

Imports form a module graph. Every pass owns a narrow invariant and validates it before the next representation is used.

02 / MEANING

HIR establishes semantics.

Name resolution, type inference, generic instantiation, and ownership checks convert parsed syntax into typed HIR. The healer applies a bounded deterministic rewrite candidate only for structured type mismatches and re-runs normal checking before producing a warning.

STRUCTURED ERRORS

Expected type, actual type, source node, and span are compiler data—not parsed strings.

LAYOUT CONTRACTS

LP64 record size, alignment, and field offsets are resolved during MIR lowering.

03 / OPTIMIZATION

MIR makes flow visible.

Basic blocks, branches, calls, loads, stores, allocations, and returns are explicit. SSA construction adds dominance-aware renaming and phi nodes. Constant folding, GVN, DCE, copy propagation, and CFG simplification operate here.

04 / REVERSIBLE OUTPUT

MIR emits typed C.

The C backend consumes optimized SSA MIR directly. Calls carry linkage, aggregate access carries byte offsets, and text literals enter a static pool. Versioned vinglish:mir comments identify every block, instruction, and terminator; host compilers discard them with no runtime cost.
